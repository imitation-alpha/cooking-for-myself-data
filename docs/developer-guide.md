# Developer Integration Guide

This guide explains how app and website developers can consume Cooking For Myself data safely and predictably.

The repository is designed for two kinds of users:

- Contributors edit human-readable source records and images.
- App and website developers consume pinned dataset releases and generated exports.

## Recommended Integration Model

Do not consume `main` directly from production apps. Pin a release tag, commit SHA, or downloaded release artifact so your app has a reproducible dataset.

Recommended flow:

1. Pick a dataset release or commit.
2. Download or vendor the data repo at that revision.
3. Run `npm test` before generating app-specific artifacts.
4. Decode records once at app or server startup.
5. Build lookup indexes by `meal.id` and `ingredient.id`.
6. Lazy-load images from the referenced paths.

Current source artifacts:

- `data/ingredients/`: ingredient source records, one JSON file per ingredient, plus `_meta.json`.
- `data/meals/`: meal source records, one JSON file per meal, plus `_meta.json`.
- `assets/ingredients/`: ingredient images.
- `assets/meals/`: meal images.

Generated artifacts:

- `dist/app-seed.json`: generated by `npm run export:app-seed`; intended for app import pipelines.

Planned generated artifacts:

- `proto/cooking_for_myself/v1/dataset.proto`: stable protobuf schema for generated clients.
- `dist/dataset.pb`: compact protobuf dataset payload.
- `dist/dataset.pb.json`: protobuf JSON representation for web and debugging workflows.

## Choosing JSON or Protobuf

Use JSON when:

- You are building web pages, admin tools, review dashboards, or import scripts.
- You want easy debugging and plain Git diffs.
- You only need a few hundred records and do not have strict startup constraints.

Use protobuf when:

- You are building a mobile app or SDK and want typed generated models.
- You want a smaller runtime payload than aggregate JSON.
- You want stable field numbers and forward-compatible decoding.
- You want generated clients for Swift, Kotlin, TypeScript, Python, Go, or other languages.

For this dataset size, images usually dominate bandwidth and storage. Protobuf is still useful because it gives application developers a stable typed interface and avoids fragile hand-written decoders.

## Data Loading Pattern

Use record IDs as the only durable references. Display text, translated fields, image paths, and sort order can change between dataset versions.

Basic runtime shape:

```ts
type DatasetIndex = {
  mealsById: Map<string, Meal>;
  ingredientsById: Map<string, Ingredient>;
};

function indexDataset(meals: Meal[], ingredients: Ingredient[]): DatasetIndex {
  return {
    mealsById: new Map(meals.map((meal) => [meal.id, meal])),
    ingredientsById: new Map(ingredients.map((ingredient) => [ingredient.id, ingredient]))
  };
}
```

When rendering a recipe:

1. Load the meal by `meal.id`.
2. Resolve each `meal.ingredients[].ingredientId` through the ingredient index.
3. Select display language from localized fields.
4. Fall back only when your product explicitly allows it.
5. Render image `path` values relative to the dataset root or your hosted asset base URL.

## Website Example

For static websites, copy a pinned dataset release into your public build or host it behind your own CDN. Avoid fetching raw GitHub URLs in production pages because rate limits and branch drift can affect users.

```ts
async function loadCookingData(baseUrl: string) {
  const seedResponse = await fetch(`${baseUrl}/dist/app-seed.json`);

  if (!seedResponse.ok) {
    throw new Error("Failed to load Cooking For Myself data");
  }

  const seed = await seedResponse.json();

  return indexDataset(seed.recipes, seed.ingredients);
}

function localized(value: Record<string, string | null>, language: string) {
  return value[language] ?? value.en ?? value.zhHant ?? "";
}
```

For images, join the dataset asset path with your hosted base URL:

```ts
function imageUrl(baseUrl: string, imagePath: string) {
  return `${baseUrl}/${imagePath}`;
}
```

## Mobile App Example

For mobile apps, prefer bundling a pinned dataset release or downloading release artifacts into an app cache. Decode at startup or during an import step, then store normalized records in your app database.

Recommended mobile behavior:

- Keep the dataset version visible in diagnostics.
- Cache decoded records by ID.
- Validate image and ingredient references before enabling a downloaded dataset.
- Keep runtime fallback behavior separate from dataset validation. The dataset may require fields that older app versions can still fallback around.
- Keep user-generated plans and favorites keyed by `meal.id`, not array position.

## Protobuf Client Generation

After the protobuf schema is published, generated clients should be created from the versioned schema, not hand-written from JSON samples.

Expected consumer flow:

1. Pin a dataset release.
2. Download `proto/cooking_for_myself/v1/dataset.proto`.
3. Generate language-specific models with `protoc`, `buf`, or your platform plugin.
4. Download `dist/dataset.pb`.
5. Decode `Dataset` with the generated client.
6. Build local indexes by record ID.

Example commands once the schema exists:

```bash
# Swift
protoc \
  --proto_path=proto \
  --swift_out=Generated \
  proto/cooking_for_myself/v1/dataset.proto

# Kotlin / Java
protoc \
  --proto_path=proto \
  --java_out=generated/java \
  proto/cooking_for_myself/v1/dataset.proto

# TypeScript with ts-proto
protoc \
  --plugin=./node_modules/.bin/protoc-gen-ts_proto \
  --ts_proto_out=generated/ts \
  --proto_path=proto \
  proto/cooking_for_myself/v1/dataset.proto
```

Swift-style decoding:

```swift
let data = try Data(contentsOf: datasetURL)
let dataset = try Cooking_ForMyself_V1_Dataset(serializedBytes: data)
let mealsByID = Dictionary(uniqueKeysWithValues: dataset.meals.map { ($0.id, $0) })
let ingredientsByID = Dictionary(uniqueKeysWithValues: dataset.ingredients.map { ($0.id, $0) })
```

TypeScript-style decoding:

```ts
import { Dataset } from "./generated/ts/dataset";

const bytes = new Uint8Array(await (await fetch("/dataset.pb")).arrayBuffer());
const dataset = Dataset.decode(bytes);
const mealsById = new Map(dataset.meals.map((meal) => [meal.id, meal]));
const ingredientsById = new Map(dataset.ingredients.map((ingredient) => [ingredient.id, ingredient]));
```

## Localization

Supported language keys:

- `zhHant`: Traditional Chinese
- `yue`: Cantonese
- `zhHans`: Simplified Chinese
- `en`: English
- `ja`: Japanese
- `ko`: Korean
- `th`: Thai
- `vi`: Vietnamese
- `id`: Indonesian
- `tl`: Tagalog / Filipino-compatible for v1

Dataset validation requires core fields to include `zhHant` and `en`. Some optional translations may be `null`.

Apps should make fallback rules explicit. For example, a Hong Kong-focused app may prefer:

```text
selected language -> yue -> zhHant -> en
```

A global English-first website may prefer:

```text
selected language -> en -> zhHant
```

Do not silently treat missing translations as reviewed translations.

## Provenance and Licensing

Every record and image includes provenance and license metadata. Preserve this metadata if you redistribute the dataset or expose it through an API.

Current license policy:

- Data records: `ODC-BY-1.0`
- Images: `CC-BY-4.0`, unless an individual image record says otherwise
- Tooling: `MIT`

Generated images must remain disclosed through the image metadata. Apps and websites may choose whether to surface that disclosure in UI, but should keep it available for audit, about pages, or dataset detail views.

## Validation

Run validation before importing a local checkout:

```bash
npm test
```

This checks schema expectations, ingredient references, image paths, language coverage, provenance, and export compatibility.

For CI in a consuming app, run this validation against the pinned dataset before generating application models or uploading release artifacts.

## Versioning Guidance

Treat dataset updates like API updates:

- Patch releases should fix metadata, translations, provenance, or images without changing IDs.
- Minor releases may add meals, ingredients, translations, images, and generated export formats.
- Major releases may remove or rename IDs, restructure records, or change protobuf fields incompatibly.

Consumers should store the dataset version together with cached records so migrations can be handled deliberately.
